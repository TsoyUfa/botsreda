# Ключевые мысли — День 2: Метод смыслокодинга

> Источник: транскрибация эфира Артемия Миллера (методология вайпкодинга: фундамент проекта, второй мозг, контекст-инжиниринг, скилы, MCP, триггеры).

---

## 1. О чём день

- Переход от «просто вайпкодить» к **системной методологии**: один короткий запрос превращается в большой контекст за счёт архитектуры папок и связей — это и есть рабочий смыслокодинг на масштабе.
- Цель: **меньше итераций и «психов в чате»**, больше предсказуемого качества за счёт заранее заложенного контекста и правил.

---

## 2. Анатомия проекта (каркас)

| Элемент | Назначение |
|---------|------------|
| **CLAUDE.md** | Стартовая инструкция агента при каждом новом диалоге: что за проект, стек, навигация, правила, язык |
| **Папка `.claude`** | Настройки Claude Code, скилы, агенты, правила (часто «невидимая» в проекте) |
| **iClone (айклон)** | Цифровая проекция **тебя**: ценности, tone of voice, принципы, фидбек, как принимаешь решения |
| **Папка бизнес** | Второй мозг **бизнеса**: продукт, аудитория, экономика, маркетинг, цели, ретроспективы |
| **Папка планов** (`plans` / планы) | Живые файлы планов и **фаз** по крупным фичам — не «план в одном чате» |
| **Мастерство** | Узкие методологии «как делают лучшие» (концентрат), не пересказ книг целиком |
| **Ретроспектива** | Зафиксированные выводы после сессий |
| **Live Matrix** | Как подключаться к **живым** цифрам (БД / API), а не хардкодить KPI в markdown |
| **MCP** | «USB-порт» к внешним сервисам (таблицы, календарь, CRM, браузер и т.д.) |
| **Триггеры** | Запуск по расписанию / вебхуку: отчёты, проверки, рутина «пока ты спишь» |

Идея: агент при задаче **ходит по графу викилинками** и подгружает только нужные файлы — экономия токенов и меньше «каши» в контексте.

---

## 3. CLAUDE.md — «трудовой договор» агента

- Подгружается **при старте каждого** нового чата (системный уровень для Claude).
- **Лимит ~150–200 строк**: не сваливать туда всё подряд; детали — в бизнес / планы / мастерство.
- Типичное содержание:
  - **Описание**: что делаем, для кого, зачем (2–3 предложения).
  - **Стек** разработки.
  - **Структура репозитория** и правила навигации («куда бежать за чем»).
  - **Роутинг** на `iClone`, бизнес, мастерство, планы.
  - **Правила**: например, не хардкодить даты/цены/настройки — брать из БД или конфигов; для крупных фич — скилл вроде **bulletproof**.
  - **Язык** ответов (например, всегда русский).
  - **Завершение диалога**: короткая рефлексия «что сделали, было → стало, решена ли задача».
- Автогенерация даёт «пустой» шаблон; **методологию** (iClone, бизнес, планы, мастерство) модель сама «не придумает» — это твоя наработка.

---

## 4. Папка планов vs Plan Mode в IDE

| | **Plan Mode (в одном чате)** | **Папка планов (файл в репо)** |
|--|------------------------------|--------------------------------|
| Жизнь плана | Пока открыт диалог | В Git, версии, история |
| Объём | Ограничен контекстом окна | Планы на **тысячи строк**, десятки фаз |
| Передача | Сложно другому агенту | Любой агент подхватывает файл |
| Итерации | Дорого по токенам | План правится в отдельных чатах по шагам |

**Практика:** один чат — сформировать план → новый чат — разбить на фазы → новый чат — проверить каждую фазу (адекватность, конфликты, безопасность, бенчмарки извне) → затем реализация.

**Правило:** **один план — одна крупная функция**; имя файла: дата + название фичи; внутри — фазы и чекбоксы.

---

## 5. Контекст-инжиниринг (главная идея вместо «промпт-инжинеринга»)

- Важно не «золотой промпт на 500 строк», а **что стабильно лежит рядом с кодом** и подмешивается по правилам.
- Пять слоёв (упрощённо): запрос → **CLAUDE.md** → **iClone** → **мастерство** (если нужно) → **бизнес** (цели, цифры, live-метрики) → ответ.
- Ошибка: запихнуть в контекст **всё подряд** (часы зумов сырым текстом) — каша и сжигание токенов. Нужно **дробить** и сохранять **смысловые выжимки** в правильные файлы.

---

## 6. iClone — не «стиль», а **как ты думаешь**

- Это не копирование лексики, а **принятие решений в твоей логике**: что ценишь, что отвергаешь, темп, амбиции, продуктовые принципы, tone of voice, **stop-слова** (что ненавидишь в текстах ИИ).
- **Фидбек-петля**: когда модель ошиблась — объяснить и попросить оформить правило в iClone; через месяц накапливаются десятки «работающих на тебя» правил.
- **Индекс.md** в iClone и в бизнесе — точка входа: карта разделов, куда проваливаться дальше.
- Один iClone на человека может **обслуживать несколько проектов** (карта ссылок на репозитории).

**Наполнение:** голосовые интервью, транскрипты эфиров/зумов, дневники, деловая переписка (с валидацией!), осторожно — автопамять из чатов (лучше проверять глазами).

---

## 7. Мастерство vs скилы

| | **Скил (skill)** | **Мастерство** |
|--|------------------|----------------|
| Суть | Повторяющийся **процесс + инструкция + опционально скрипты** (PDF, PPTX, frontend-стек и т.д.) | **Методология предмета** — сжатый концентрат лучших практик |
| Пример | «Сделай PDF из HTML по шаблону» | «Как пишет сценарии автор X» — выжатые ключевые методы, не книга целиком |
| Источник | Свои/чужие репозитории скилов | Транскрипты, курсоры по 1–3 ключевым методам на автора |

Принцип мастерства: **один автор предмета — один файл**, связи викилинками; цель — **мало килобайт концентрата**, а не энциклопедия.

---

## 8. Папка бизнес — второй мозг компании

- Markdown + викилинки рядом с кодом: **одно место правды**, снимает нагрузку с головы («почему в марте отказались от рассрочки?» → ответ из файла).
- Структура через **индекс**, продукт, аудитория (аватар, боли, путь клиента), экономика, маркетинг, цели (год / квартал / месяц / неделя), KPI.
- **Не захламлять:** сырые материалы — в папку вроде `raw`, оттуда — извлечь ключевое и разложить по местам с **ручной валидацией**.
- **Обновление:** в конце дня — «пройди git-историю / что сделали; что важно обновить в бизнес?»; анализ лендинга / сайта → вынести смыслы в нужные файлы.
- **Персональные данные:** не слать в облако модели; при необходимости — **read-only** к БД, обезличенные API, схема в Live Matrix.

---

## 9. Золотые правила экономии контекста и денег

1. **Один чат — одна задача.** Закончил этап — закрыл чат, открыл новый (иначе в историю тянется всё, токены улетают).
2. При переполнении: «**напиши промпт для соседнего чата**», чтобы продолжить без потери смысла (альтернатива — compact/clear, но автор чаще использует перенос промптом).
3. **Контекст >60–70%** — сигнал сменить чат.
4. **По возможности не править руками** файлы, которые должен вести агент — иначе не угонишь за скоростью среды; голос + итерации правок.
5. **Думать дольше, кодить короче:** часы проработки архитектуры и плана vs меньше времени на «перепиливание» кода.

## 10. Методология в 6 шагах (цикл по задаче)

Упрощённая цепочка от идеи до результата:

1. Человеческий запрос «что хочу».
2. Уточняющие вопросы модели.
3. Поиск внешних решений / методологий / бенчмарков.
4. Критика и доказательства «почему это сработает / где риск разориться через год».
5. План (в файле) → фазы → реализация в **чистом** чате по плану.
6. Итерации, скриншоты, правки; важное из правок — обратно в бизнес / iClone.

Дополнительно по фиче: **research → план → глубокая проверка плана → фазы → код → review / поиск своих ошибок**; скилл **«10 причин облажаться»** для критичных задач.

## 11. Скилы — практика

- Установка **по ссылке на репозиторий**: «установи скилл в проект / глобально».
- Полезные базовые: **frontend-design**, **skill-creator**, **PDF**, **PPTX** (у автора часто глобально).
- **Аудит перед установкой** чужого скилла: README, опасные операции (env, внешние URL, удаление, бинарники) — вердикт безопасно / нет.
- Скилл: `SKILL.md` + опционально `scripts/`, `templates/`; **прогрессивная подгрузка** описания.

---

## 12. MCP и CLI

- **MCP** удобен для интеграций, но **дорог по токенам** (пример порядка: много MCP — десятки тысяч токенов на запрос); включать по необходимости.
- **CLI-утилиты** (open source): мало нагрузки на контекст — короткий вызов, тяжёлая работа локально (парсинг, медиа и т.д.).

---

## 13. Субагенты и параллель

- Оркестратор в одном чате может запустить **несколько субагентов** на разные фазы или исследования, потом — критик и сводка.
- **Риск:** много агентов без дисциплины ломают код; на старте важнее **один** стабильный агент, чем «60 супер-агентов».

## 15. Расширение Claude в Chrome

- Для задач **в браузере** (регистрации, настройки кабинетов, метрики, формы): агент управляет страницей, скриншоты, «мышь», сценарий из VS Code / Claude Code через промпт в расширение.

---

## 16. Триггеры

**Три кирпича:**

1. **Когда** — cron / планировщик (например, каждый день в 7:00).
2. **Сценарий** — файл-инструкция: куда зайти, что прочитать (бизнес, мастерство, поддержка), что сгенерировать.
3. **Куда доставить** — Telegram-бот, email, push и т.д.

Примеры: утренний финансовый разбор; разбор тикетов поддержки + предложение фиксов; мониторинг конкурентов; обновление отчётов с **live**-цифрами.

**Инфра:** для круглосуточной работы — ПК без сна **или** Claude Code на **удалённом сервере** + бот.

---

## 17. Безопасность в bypass / настройках

- Запреты: полное удаление, утечка секретов, отправка ключей на внешние URL, чтение `.env` без нужды.
- Осторожность с **фишинговыми** скиллами из интернета.

---

## 18. Мелкие сильные приёмы

- **Звук / уведомление** при завершении работы или запросе разрешений в VS Code — не пропускать «готовность» агента.
- **Скриншот в буфер** (горячие клавиши под себя) + вставка в чат для правок UI.
- **WhisperFlow** — голос; в инструкции дня 1/2 — локальный бесплатный вариант через скилл сообщества.

---

## 19. Ключевая мысль дня

> **Промпт-инжинеринг уступает место контекст-инжинерингу:** качество и скорость растут не от длины одного сообщения, а от **архитектуры знаний** (CLAUDE.md + iClone + бизнес + мастерство + планы + живые метрики) и дисциплины **один чат — одна задача**.

Домашний каркас к следующему шагу: **CLAUDE.md**, папка **планов**, зачатки **iClone**, **бизнес**, немного **мастерства**, понимание **контекст-инжиниринга** и **триггеров**; день 3 — деплой, git, живой проект по методологии.
