# Принципы кода и автоматизации

## Главный принцип

Технологии нужны только там, где они дают бизнес-эффект: продажи, скорость, контроль, снижение ручной работы, качество данных или масштабирование.

## Перед автоматизацией

Сначала описать процесс:

- входящие данные;
- участники;
- шаги;
- точки принятия решений;
- результат;
- узкие места;
- частота повторения;
- кто будет пользоваться.

## Правила

- Не хранить секреты, токены и пароли в заметках.
- Не хардкодить важные даты, цены, KPI и токены.
- Перед сложной реализацией делать план.
- Использовать простые интеграции, если они решают задачу.
- CRM, API и таблицы использовать как практичный стек автоматизации.
- Избегать технического театра, если бизнес-результат не меняется.
- Автоматизировать после 3+ повторов процесса или при явной экономике.

## Практичный стек

- Obsidian - база знаний и второй мозг.
- Cursor - работа с файлами, инструкциями, кодом и AI-логикой.
- CRM - сделки, статусы, клиенты, источники.
- API - связь с CRM, банками, застройщиками, таблицами.
- Telegram - коммуникации, боты, уведомления.
- Google Sheets / Яндекс Таблицы - быстрые MVP и расчёты.

# Принципы кода и автоматизаций

## Общий подход

Даже если задача не про классическую разработку, агент должен работать как продуктовый инженер: сначала цель, процесс и бизнес-результат, потом инструмент.

## Правила

- Не хардкодить важные даты, цены, KPI, токены и секреты.
- Не хранить пароли, ключи API и приватные данные в заметках.
- Перед сложной реализацией делать короткий план.
- Автоматизировать только понятный процесс.
- Использовать простые интеграции, если они решают задачу.
- CRM, API, таблицы, Telegram и Obsidian использовать как практичный стек.
- Не делать "технический театр", если бизнес-результат не меняется.

## Перед автоматизацией ответить

- Какой процесс автоматизируем?
- Где сейчас ручной труд или хаос?
- Кто пользователь?
- Что считается успешным результатом?
- Какая метрика улучшится?
- Можно ли сначала сделать полуавтоматически?

## Для Cursor и Obsidian

- Не менять старые заметки без прямого запроса.
- Новые структуры создавать рядом, с понятными названиями.
- Если информации не хватает, ставить `[добавить инфу: ...]`.
- Для больших проектов создавать `INDEX.md` или карту входа.
- Не плодить файлы без маршрутизации и следующего действия.
