## Введение

### Что такое модели в Cursor

Cursor предоставляет доступ к различным языковым моделям ИИ, каждая из которых оптимизирована для определенных типов задач. Правильный выбор модели напрямую влияет на:
- **Качество результата** — насколько хорошо модель справится с задачей
- **Скорость работы** — как быстро вы получите ответ
- **Стоимость** — сколько будет стоить использование

### Что означает иконка "мозга" 🧠

**Модели С иконкой мозга:**
- Обладают расширенными возможностями рассуждения (Thinking mode)
- Лучше справляются со сложными многошаговыми задачами
- Могут выполнять глубокий анализ перед выполнением задачи
- Требуют больше ресурсов и времени
- Стоят дороже

**Модели БЕЗ иконки мозга:**
- Более легкие и быстрые модели
- Подходят для простых и рутинных задач
- Быстрее отвечают
- Дешевле в использовании
- Меньше потребляют ресурсов

### Принципы выбора модели

1. **Начинайте с простого** — для обычных задач используйте модели без иконки мозга
2. **Переходите к сложному** — для многошаговых задач переключайтесь на модели с мозгом
3. **Учитывайте бюджет** — для больших проектов используйте более дешевые модели
4. **Тестируйте** — пробуйте разные модели для своих задач, чтобы найти оптимальную

## Описание моделей

### 1. Composer 1

**Иконка мозга:** ❌ Нет

#### Характеристики

- **Скорость:** ⚡⚡⚡⚡ Очень быстрая (4x быстрее Opus 4.5)
  - Скорость вывода: ~250 токенов/секунду
  - Среднее время выполнения задачи: 8.9 минут (vs 12.6 минут у GPT-5)
- **Контекстное окно:** 200K токенов
- **Стоимость:** Средняя (специфичная для Cursor)
- **Производительность:** Оптимизирована для работы с реальными кодовыми базами

#### Сильные стороны

✅ **Мультифайловый рефакторинг**
- Отлично работает с несколькими файлами одновременно
- Понимает связи между файлами в проекте
- Эффективно выполняет рефакторинг больших участков кода

✅ **Скорость итераций**
- Идеальна для быстрого прототипирования
- Когда важна скорость больше, чем идеальное качество
- Для задач, где нужно много раз переделывать

✅ **Работа с реальными проектами**
- Лучше других моделей понимает структуру реальных проектов
- Учитывает контекст всего проекта, а не только текущего файла

#### Слабые стороны

❌ **Не для сложной логики**
- Может упускать детали в очень сложных алгоритмах
- Не лучший выбор для глубокой архитектурной работы

❌ **Ограниченное рассуждение**
- Нет расширенного режима Thinking
- Может не справиться с многошаговыми задачами, требующими планирования

#### Практические примеры использования

**Сценарий 1: Рефакторинг компонентов React**
```
Задача: Переименовать компонент UserCard в ClientCard 
во всех файлах проекта (15 файлов)

Промпт: "Переименуй компонент UserCard в ClientCard 
во всех файлах проекта, включая импорты и экспорты"

Почему Composer 1: Быстро обработает все файлы, 
понимая связи между ними
```

**Сценарий 2: Автоматизация для недвижимости**
```
Задача: Создать систему автоматической отправки 
уведомлений в Telegram при новых заявках из CRM

Промпт: "Создай интеграцию n8n workflow, который 
читает новые заявки из AMOCRM API и отправляет 
уведомления в Telegram бот"

Почему Composer 1: Нужна быстрая итерация для 
прототипа, можно доработать позже
```

**Сценарий 3: Массовое обновление кода**
```
Задача: Обновить все вызовы устаревшего API 
на новый формат

Промпт: "Замени все вызовы старого API endpoint 
/api/v1/clients на /api/v2/clients во всех файлах"

Почему Composer 1: Отлично для массовых изменений 
в нескольких файлах одновременно
```

### 2. Claude Opus 4.5

**Иконка мозга:** ✅ Да (расширенное рассуждение)

#### Характеристики

- **Скорость:** ⚡⚡ Средняя
- **Контекстное окно:** 200K токенов
- **Стоимость:** $5 вход / $25 выход за 1M токенов (снижение на 67% от предыдущей версии)
- **Производительность:** 
  - SWE-Bench Verified: >80% (первая модель, преодолевшая этот порог)
  - Лучшая модель для кодинга по большинству бенчмарков

#### Сильные стороны

✅ **Лучшее качество кода**
- Высокое качество и консистентность генерируемого кода
- Отличное понимание лучших практик программирования
- Следует принципам SOLID, KISS и другим паттернам

✅ **Сложная отладка**
- Отлично находит и исправляет сложные баги
- Понимает причинно-следственные связи в коде
- Может анализировать логи и трассировки

✅ **Архитектурное планирование**
- Помогает проектировать архитектуру приложений
- Учитывает масштабируемость и поддерживаемость
- Предлагает несколько вариантов с анализом плюсов/минусов

✅ **Агентные workflow**
- Отлично работает в режиме агента с множеством шагов
- Может планировать сложные многошаговые задачи
- Использует расширенное рассуждение (Thinking mode)

✅ **Аудит безопасности**
- Находит уязвимости в коде
- Предлагает безопасные альтернативы
- Понимает риски безопасности

#### Слабые стороны

❌ **Медленнее, чем Composer**
- Не лучший выбор, когда важна скорость
- Для простых задач может быть избыточен

❌ **Ограниченный контекст**
- 200K токенов может быть недостаточно для очень больших проектов
- Для анализа огромных кодовых баз лучше Sonnet 4.5 или Gemini

#### Практические примеры использования

**Сценарий 1: Разработка системы анализа звонков**
```
Задача: Создать систему автоматического анализа 
звонков с клиентами для застройщика

Промпт: "Спроектируй архитектуру системы анализа 
звонков: транскрибация через AssemblyAI, анализ 
через GPT, интеграция с AMOCRM через n8n. 
Учти масштабируемость до 2000 звонков/месяц"

Почему Opus 4.5: Нужна архитектура, планирование, 
учет множества интеграций
```

**Сценарий 2: Отладка сложного бага**
```
Задача: Найти причину падения системы при 
обработке больших объемов данных

Промпт: "Проанализируй логи и код обработки данных. 
Система падает при обработке более 10,000 записей. 
Найди причину и предложи решение"

Почему Opus 4.5: Требуется глубокий анализ, 
понимание причинно-следственных связей
```

**Сценарий 3: Рефакторинг legacy-кода**
```
Задача: Переписать старый код на современные 
практики с сохранением функциональности

Промпт: "Рефакторинг этого модуля: примени SOLID 
принципы, улучши читаемость, добавь типизацию. 
Сохрани всю функциональность"

Почему Opus 4.5: Нужно высокое качество кода, 
понимание паттернов, сохранение логики
```

### 3. Claude Sonnet 4.5

**Иконка мозга:** ✅ Да (расширенное рассуждение) и ❌ Нет (базовая версия)

#### Характеристики

**Версия С мозгом:**
- **Скорость:** ⚡⚡⚡ Быстрая
- **Контекстное окно:** 1M токенов (в 5 раз больше, чем у Opus!)
- **Стоимость:** $3 вход / $15 выход за 1M токенов (в 2 раза дешевле Opus)
- **Производительность:** Хорошая, но немного ниже Opus на сложных задачах

**Версия БЕЗ мозга:**
- **Скорость:** ⚡⚡⚡⚡ Очень быстрая
- **Контекстное окно:** 1M токенов
- **Стоимость:** Еще дешевле
- **Производительность:** Базовая, для простых задач

#### Сильные стороны

✅ **Большой контекст**
- Может анализировать огромные кодовые базы целиком
- Работает с проектами в сотни тысяч строк кода
- Не теряет контекст при работе с большими файлами

✅ **Бюджетный вариант**
- В 2 раза дешевле Opus 4.5
- Отличное соотношение цена/качество
- Подходит для больших проектов с ограниченным бюджетом

✅ **Хорошее качество**
- Качество кода близко к Opus на большинстве задач
- Отлично для ежедневной разработки
- Хорошо понимает контекст проекта

#### Слабые стороны

❌ **Может использовать устаревшие данные**
- Иногда полагается на обученные данные вместо ваших инструкций
- Нужно четко указывать требования

❌ **Немного слабее Opus**
- На очень сложных задачах может уступать Opus
- Для критически важного кода лучше Opus

#### Практические примеры использования

**Сценарий 1: Анализ большой кодовой базы**
```
Задача: Проанализировать весь проект автоматизации 
(500+ файлов) и найти все места интеграции с CRM

Промпт: "Проанализируй весь проект и найди все 
места, где происходит интеграция с AMOCRM. 
Составь список файлов и методов"

Почему Sonnet 4.5: Нужен большой контекст для 
анализа всего проекта
```

**Сценарий 2: Ежедневная разработка**
```
Задача: Добавить новую функцию в существующий проект

Промпт: "Добавь функцию экспорта данных в Excel 
в модуль отчетности. Используй существующие паттерны"

Почему Sonnet 4.5: Хорошее качество за разумную цену, 
достаточно для обычных задач
```

**Сценарий 3: Документирование проекта**
```
Задача: Создать документацию для всего проекта

Промпт: "Проанализируй весь проект и создай 
подробную документацию: архитектура, основные 
модули, API endpoints, примеры использования"

Почему Sonnet 4.5: Нужен большой контекст, 
не требуется максимальное качество кода
```

## Рекомендации по использованию

### Когда использовать модели С иконкой мозга 🧠

Используйте модели с расширенным рассуждением для:

1. **Сложных многошаговых задач**
   - Когда нужно планировать несколько шагов
   - Задачи, требующие анализа перед выполнением
   - Архитектурные решения

2. **Критически важного кода**
   - Production-код высокого качества
   - Системы обработки платежей
   - Безопасность и защита данных

3. **Глубокой отладки**
   - Сложные баги, которые трудно найти
   - Анализ логов и трассировок
   - Понимание причинно-следственных связей

4. **Агентных workflow**
   - Автоматизация с множеством шагов
   - Системы, которые принимают решения
   - Интеграции с внешними API

### Когда достаточно моделей БЕЗ иконки мозга

Используйте более легкие модели для:

1. **Простых задач**
   - Добавление функций
   - Исправление простых багов
   - Рефакторинг отдельных файлов

2. **Рутинной работы**
   - Обновление документации
   - Переименование переменных
   - Форматирование кода

3. **Быстрых итераций**
   - Когда важна скорость
   - Прототипирование
   - Эксперименты

### Стратегия переключения между моделями

**Подход "Снизу вверх":**
1. Начните с простой модели (Sonnet без мозга или Gemini Flash)
2. Если результат неудовлетворителен — переключитесь на более мощную
3. Для критических задач сразу используйте Opus 4.5

**Подход "Сверху вниз":**
1. Начните с Opus 4.5 для важных задач
2. Используйте более простые модели для доработок
3. Оптимизируйте затраты на простых задачах

**Гибридный подход (рекомендуется):**
- **Composer 1** — для рефакторинга и работы с несколькими файлами
- **Sonnet 4.5** — для ежедневной разработки
- **Opus 4.5** — для сложных задач и архитектуры
- **Gemini Flash** — для больших объемов и анализа проектов
- **GPT-5.2 Codex** — для математики и абстрактных задач

### Оптимизация затрат

1. **Используйте дешевые модели для простых задач**
   - Gemini Flash для рутины
   - Sonnet 4.5 для обычной разработки

2. **Экономьте на больших объемах**
   - Gemini Flash для анализа больших проектов
   - Sonnet 4.5 вместо Opus для больших задач

3. **Инвестируйте в качество для важного**
   - Opus 4.5 для критического кода
   - Не экономьте на безопасности

4. **Комбинируйте модели**
   - Быстрый прототип на Composer → доработка на Opus
   - Анализ на Gemini → реализация на Sonnet

### Как тестировать модели для своих задач

1. **Создайте тестовый набор задач**
   - Типичные задачи из вашей работы
   - Разной сложности
   - С известными решениями

2. **Протестируйте каждую модель**
   - Дайте одинаковые промпты разным моделям
   - Сравните результаты по качеству, скорости, стоимости
   - Зафиксируйте результаты

3. **Создайте свою таблицу выбора**
   - Для каждого типа задач — оптимальная модель
   - Учитывайте ваш контекст и требования

### Типичные ошибки при выборе модели

❌ **Ошибка 1: Всегда использовать самую мощную модель**
- **Проблема:** Ненужные затраты на простые задачи
- **Решение:** Используйте Opus только для сложных задач

❌ **Ошибка 2: Игнорировать контекстное окно**
- **Проблема:** Модель теряет контекст на больших проектах
- **Решение:** Используйте Sonnet или Gemini для больших проектов

❌ **Ошибка 3: Не учитывать скорость**
- **Проблема:** Медленная работа на простых задачах
- **Решение:** Используйте Composer для быстрых итераций

❌ **Ошибка 4: Не тестировать альтернативы**
- **Проблема:** Использование неоптимальной модели
- **Решение:** Пробуйте разные модели для своих задач

❌ **Ошибка 5: Не комбинировать модели**
- **Проблема:** Использование одной модели для всего
- **Решение:** Используйте разные модели для разных этапов

### Чек-лист выбора модели

Перед выбором модели ответьте на вопросы:

- [ ] Насколько сложна задача? (простая → Sonnet без мозга, сложная → Opus)
- [ ] Важна ли скорость? (да → Composer/Gemini, нет → Opus)
- [ ] Какой размер проекта? (<200K → любая, >200K → Sonnet/Gemini)
- [ ] Какой бюджет? (ограничен → Gemini/Sonnet, неограничен → Opus)
- [ ] Нужно ли абстрактное мышление? (да → GPT-5.2 Codex)
- [ ] Нужна ли математика? (да → GPT-5.2 Codex)
- [ ] Сколько файлов затронуто? (много → Composer)
- [ ] Критически ли важен код? (да → Opus 4.5)
