# 5. MVP-2 — Клиенты, сделки, монетки, рейтинг

## 5.1. Скоуп MVP-2

**Добавляем к MVP-1:**
- Топ-100 / мини-CRM (карточки клиентов, воронка, заметки).
- Сделки (создание, подтверждение, статусы).
- Монетки и шкала бонусов.
- Рейтинг потока.
- Связки: ДЗ Недели 4 (диагностика клиента) и Недели 5 (финмодель) подвязываются к карточке клиента.

**Вне скоупа MVP-2:**
- Telegram-бот.
- Генераторы PDF.
- Интеграция с CRM агентства (если есть).
- Командные челленджи.

**Цель MVP-2:** дашборд становится **рабочим инструментом**, а не «учебным классом» — новички ведут реальных клиентов и сделки.

## 5.2. Новые экраны

### 5.2.1. `/clients`

**Назначение:** список клиентов агента (Топ-100 + новые лиды).

UI:
- Шапка: общий счётчик контактов, кнопка «Добавить контакт», поиск.
- Фильтры по статусу воронки: `new`, `dialog`, `selection_sent`, `show`, `reservation`, `deal`, `paid`, `lost`.
- Канбан-вид (опция) и табличный вид (по умолчанию).
- В табличном виде колонки: имя, телефон, источник, статус, дата последнего касания, кол-во касаний, оффер.

Действия из списка:
- Массовое добавление (Топ-100): импорт из CSV или вставка списка имён.
- Быстрый `+1 касание` прямо из строки.

### 5.2.2. `/clients/[id]`

**Назначение:** карточка клиента.

Секции:
1. **Шапка:** ФИО, телефон (маска), источник, тег статуса воронки.
2. **Push / Pull / Anxiety / Habit** — четыре поля для сил прогресса (на пять записей каждое).
3. **Диагностика** — 10 вопросов из курса Недели 4. Заполняется агентом, привязывается к ДЗ.
4. **Лента касаний** — все `actions` с этим клиентом + кнопка `+1` (тип на выбор).
5. **Подборки и финмодели** — список артефактов типа `selection` и `finmodel`, привязанных к клиенту.
6. **Фиксация уникальности** — отдельная карточка со статусом (учебная / реальная / отправлена / подтверждена / сгорела).
7. **Сделка** — если есть, ссылка на карточку сделки.
8. **Заметки** — свободный текст с историей.

UX-фишка: при заполнении диагностики и фиксации уникальности **автоматически предлагается перевести связанное ДЗ в `review`**.

### 5.2.3. `/deals`

**Назначение:** список сделок агента.

UI:
- Карточки сделок с цветовой индикацией статуса (`reservation` / `contract` / `paid`).
- Сводка сверху: количество монеток, сколько до следующего бонуса, количество сделок этого месяца.
- Кнопка «Зафиксировать новую сделку».

### 5.2.4. `/deals/[id]`

**Назначение:** карточка сделки.

Поля:
- Клиент (ссылка на `/clients/[id]`).
- ЖК (свободный текст или из базы знаний).
- Сумма сделки (₽).
- Комиссия (₽).
- Дата брони / договора / оплаты.
- Статус.
- Подтверждающий (РОП / наставник).
- Документы (загрузка скринов).

Действия:
- «Перевести в статус Договор» (агент).
- «Перевести в Деньги получены» — только наставник/РОП. После подтверждения автоматически: 1 монетка + 1 000 XP, проверка ачивок (`closer`, `triple`, `jk_ambassador`), пересчёт бонусов.

### 5.2.5. `/coins`

**Назначение:** баланс монеток и шкала бонусов.

UI:
- Большой счётчик монеток.
- Прогресс-бар до следующего бонуса.
- Лента «как заработал»: список сделок и спец-источников.
- Карточка «Мой статус по комиссии»: текущая ставка (например, 50/50), целевая (65/35), сколько монеток до перехода.

### 5.2.6. `/leaderboard`

**Назначение:** рейтинг потока.

UI:
- Переключатели периода: «эта неделя / месяц / весь поток».
- Переключатель метрики: XP, касания, диалоги, сделки, монетки.
- Топ-3 закреплены сверху с аватарами.
- Список 4-100 в виде карточек.
- Ваше место всегда видно — закреплено в нижней части экрана.

Анонимизация:
- Ник + аватар, без email и телефона.
- Если поток < 5 человек — рейтинг скрыт (нет смысла соревноваться).

## 5.4. Новые сущности и поля

```sql
-- Расширение clients
alter table clients add column funnel_status text not null default 'new'
  check (funnel_status in ('new','dialog','selection_sent','show','reservation','deal','paid','lost'));
alter table clients add column source text;
alter table clients add column offer_code text;
alter table clients add column last_action_at timestamptz;
alter table clients add column notes_md text;

-- Уникальность
create table uniqueness_fixations (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references users(id),
  client_id uuid not null references clients(id),
  jk text,
  status text not null check (status in ('draft','sent','confirmed','expired','lost')),
  developer_response_at timestamptz,
  is_demo boolean default false,
  created_at timestamptz not null default now()
);

-- Финмодель
create table finmodels (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references users(id),
  client_id uuid references clients(id),
  data jsonb not null,
  status text not null default 'draft',
  created_at timestamptz not null default now()
);

-- Бонусы (история выдачи)
create table bonus_grants (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references users(id),
  bonus_code text not null,
  coins_required int not null,
  amount_rub numeric(12,2),
  granted_at timestamptz not null default now(),
  paid_at timestamptz
);

-- Кешированные агрегаты для рейтинга (опционально, для производительности)
create materialized view leaderboard_weekly as
  select
    u.id as user_id,
    u.cohort_id,
    coalesce(sum(case when xe.created_at > now() - interval '7 days' then xe.amount end), 0) as xp_week,
    coalesce(count(distinct case when a.occurred_at > now() - interval '7 days' then a.id end), 0) as actions_week,
    coalesce(count(distinct case when d.confirmed_at > now() - interval '7 days' then d.id end), 0) as deals_week
  from users u
  left join xp_events xe on xe.user_id = u.id
  left join actions a on a.user_id = u.id
  left join deals d on d.user_id = u.id and d.status = 'paid'
  group by u.id, u.cohort_id;
```

Refresh: каждые 15 минут через Edge Function.

## 5.6. Новые ачивки и условия

| Код | Когда срабатывает |
|-----|-------------------|
| `closer` | первая запись в `coins` для пользователя |
| `triple` | 3 записи в `coins` за последние 30 дней |
| `jk_ambassador` | ≥ 3 сделок в одном `jk` за 90 дней + лидер потока по этому ЖК |
| `top_100_full` | 100 уникальных записей в `clients` с `user_id` |
| `voice_of_market` | 10 записей в `kb_objections` (новая таблица в MVP-2) |

## 5.7. Acceptance criteria MVP-2

- [ ] Новичок создаёт клиентов и ведёт их по воронке.
- [ ] Клиент привязывается к диагностике (Неделя 4) и финмодели (Неделя 5) автоматически переводит ДЗ в `review`.
- [ ] Агент создаёт сделку, наставник подтверждает «Деньги получены», после чего: монетка зачислена, XP начислены, ачивки выданы, бонус записан в историю.
- [ ] Шкала бонусов (3/5/10/20 монеток) корректно срабатывает.
- [ ] Рейтинг потока работает в 3 периодах × 5 метриках.
- [ ] Anti-cheat: нельзя сделать 2 монетки за одну сделку (уникальность по `deal_id` в `coins`).
- [ ] e2e-сценарии:
  1. Создание клиента → 5 касаний → диагностика → подборка → бронь → договор → оплата → монетка.
  2. 3 монетки → автоматически всплывает плашка «Получен бонус Стартер +5 000 ₽».

## 5.9. Риски и решения

| Риск | Решение |
|------|---------|
| Конфликт с существующей CRM агентства | На MVP-2 не интегрируемся, в Hub есть напоминание «не забудьте перенести в CRM» |
| Кто подтверждает сделку, если несколько РОПов | Поле `mentor_id` у клиента + любая роль `admin` |
| Накрутка сделок на учебные ЖК | В UI новый агент не может создать сделку без минимум 5 касаний по клиенту и подтверждения наставника |
| Рейтинг демотивирует отстающих | Закрепляем своё место «вокруг», показываем 2 человек выше и ниже, чтобы был достижимый ориентир |
